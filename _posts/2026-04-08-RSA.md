RSA는 가장 대표적인 공개키 암호 방식이다. 공개키로 암호화하고 개인키로 복호화할 수 있고, 반대로 개인키로 서명하고 공개키로 검증할 수도 있다.

이 글은 RSA를 빠르게 다시 떠올릴 수 있게 핵심만 정리한 메모다.

## 1. 큰 흐름

RSA에서 핵심은 두 가지다.

- 공개키: 누구나 알아도 되는 값
- 개인키: 소유자만 알아야 하는 값

공개키로 평문을 암호문으로 만들고, 개인키로 다시 평문을 복원한다.

## 2. 키 생성

RSA 키 생성 순서는 보통 아래와 같다.

1. 큰 소수 `p`, `q`를 고른다.
2. `n = p * q`
3. `phi(n) = (p - 1) * (q - 1)`
4. `1 < e < phi(n)` 이고 `gcd(e, phi(n)) = 1` 인 `e`를 고른다.
5. `d * e ≡ 1 (mod phi(n))` 를 만족하는 `d`를 구한다.

여기서:

- 공개키는 `(n, e)`
- 개인키는 `(n, d)`

즉 공개키는 모두에게 공개되고, 개인키는 `d`와 사실상 `p`, `q`를 알고 있는 사람만 안전하게 유지할 수 있다.

## 3. 암호화와 복호화

평문을 `m`, 암호문을 `c`라고 하면:

- 암호화: `c = m^e mod n`
- 복호화: `m = c^d mod n`

Python에서는 보통 이렇게 계산한다.

```python
c = pow(m, e, n)
m = pow(c, d, n)
```

## 4. 왜 복호화가 되는가

핵심은 `d * e ≡ 1 (mod phi(n))` 이다.

즉 어떤 정수 `k`에 대해:

`d * e = 1 + k * phi(n)`

가 성립한다. 그래서 복호화할 때:

`m^(ed) = m^(1 + k * phi(n))`

꼴이 되고, 오일러 정리를 이용하면 원래의 `m`으로 돌아오게 된다.

직관적으로 보면:

- `e`는 잠그는 지수
- `d`는 다시 푸는 지수

라고 생각하면 된다.

## 5. 작은 예시

자주 나오는 장난감 예시는 다음과 같다.

- `p = 61`
- `q = 53`
- `n = 3233`
- `phi(n) = 3120`
- `e = 17`
- `d = 2753`

평문 `m = 65`를 암호화하면:

- `c = 65^17 mod 3233 = 2790`

다시 복호화하면:

- `2790^2753 mod 3233 = 65`

실제 서비스에서는 이런 작은 수를 절대 쓰지 않고, 훨씬 큰 소수를 사용한다.

## 6. 전자서명

RSA는 암호화뿐 아니라 서명에도 쓴다.

- 서명: `s = H(m)^d mod n`
- 검증: `H(m)`와 `s^e mod n`을 비교

여기서 `H(m)`는 메시지 자체가 아니라 해시값이다. 실제 구현에서 원문 전체에 바로 RSA를 적용하는 식으로 생각하면 틀린다.

## 7. 실무에서 중요한 점

가장 중요한 건 교과서식 RSA를 그대로 쓰면 안 된다는 점이다.

- 암호화에는 보통 `RSA-OAEP`
- 서명에는 보통 `RSA-PSS`

같은 패딩 방식이 필요하다.

패딩이 없으면:

- 같은 메시지가 항상 같은 암호문으로 나올 수 있고
- 구조가 노출될 수 있고
- 여러 고전적 공격이 가능해진다

즉 `pow(m, e, n)`만으로 끝나는 RSA는 공부용 예제일 뿐, 실무 구현으로 보면 위험하다.

## 8. CTF에서 자주 보는 체크포인트

CTF에서 RSA 문제가 나오면 보통 아래를 먼저 본다.

- `n`이 인수분해 가능한가
- `p`, `q`가 너무 가깝지 않은가
- `e`가 너무 작고 패딩이 없는가
- 여러 문제에서 같은 `n`이나 같은 소수를 재사용했는가
- `dp`, `dq`, `phi(n)` 같은 값이 일부라도 유출됐는가
- 개인키 관련 부등식이 작아서 Wiener 공격 조건에 걸리는가

문제 풀이에서는 수학 자체보다도 "무엇이 잘못 설정되었는지"를 찾는 감각이 더 중요하다.

## 9. 마무리

RSA를 짧게 요약하면:

- 소수 두 개로 `n`을 만든다
- 공개키 `(n, e)`와 개인키 `(n, d)`를 만든다
- 암호화는 `m^e mod n`
- 복호화는 `c^d mod n`
- 실제 사용에서는 반드시 적절한 패딩이 필요하다

RSA는 구조 자체는 단순하지만, 구현과 운용에서 실수가 생기면 바로 약점으로 이어진다. 그래서 원리 이해와 함께 "어떤 조건이 깨지면 공격이 되는지"를 같이 보는 게 중요하다.
